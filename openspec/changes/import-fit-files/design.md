## Context

The app currently only supports activity data from Strava API. Athletes with non-Strava devices (Garmin, Wahoo, Coros, etc.) cannot import their `.fit` files. The Dexie database (`AthleteInsightDB`) stores activities in a `activities` table (keyed by Strava's numeric `id`) + `activityDetails` with stream data, plus FK tables (`segments`, `segmentEfforts`, `routeGroups`, `routeActivities`, `activitySegments`) that reference `activityId: number`. All current routes and components assume Strava-sourced data with numeric IDs. No PWA service worker or share target exists.

The primary key type mismatch (Strava uses `number`, FIT imports need strings) makes a unified table with a common key type desirable. This is a good opportunity to migrate to a source-agnostic schema.

## Goals / Non-Goals

**Goals:**
- Parse raw `.fit` binary files and extract activity summaries + stream data
- Provide a dedicated import page with drag-and-drop support for batch uploading
- Create a unified `allActivities` table with UUID primary keys for all activities (Strava + device)
- Migrate all existing Strava data (activities, details, segments, efforts, routes) to the unified schema
- Register PWA `share_target` so `.fit` files shared from other apps route to the import page
- Show all activities in the existing activity list (`Activities.tsx`) with a `source` badge

**Non-Goals:**
- Editing/deleting individual imported files after upload (browser refresh clears temp state)
- Strava upload integration (push FIT data to Strava)
- Real-time sync with device platforms (Garmin Connect, Wahoo, etc.)
- Offline service worker caching of the app (out of scope — only share target is needed)

## Decisions

| Decision | Choice | Rationale | Alternatives Considered |
|---|---|---|---|
| FIT parsing library | `fit-file-parser` (npm) | Most maintained JS FIT parser, handles all common FIT message types (records, sessions, laps), MIT license | `@garmin/fitsdk` (heavier, Garmin-specific); custom binary parser (too error-prone) |
| Storage model | Unified `allActivities` + `allActivityDetails` tables with `id: string` (UUID) and `source: 'strava' | 'device'` field. Strava numeric ID stored as `externalId?`. FK tables (`segments`, `segmentEfforts`, `routeGroups`, `routeActivities`, `activitySegments`) mirror the change: `activityId: string` | Single key type everywhere, source-agnostic queries, clean model for future import types. Migration is one-time cost. | Separate `deviceActivities` table (dual query paths, never cleans up legacy); `source` column on existing tables (number|string union types everywhere) |
| Batch import flow | Read all files client-side via FileReader, parse sequentially with progress tracking, bulk-write to Dexie in a single transaction per file | Simple, no server needed, user sees real-time progress | Web Worker per file (overkill for typical < 50 MB FIT files); server-side upload (adds infra cost) |
| PWA share target | `manifest.json` `share_target` + service worker `fetch` handler | Standard PWA approach supported on Chrome/Android; SW intercepts POST from share and redirects to import page with file data | `launchQueue` API only (inconsistent browser support); Custom URL scheme (not portable) |
| Activity list integration | Read from `allActivities` table, filter/sort in query, add `source` badge | Single query returns all activities, no merging | Querying two separate tables and merging in JS (more code, slower) |
| FIT file reading | `ArrayBuffer` via `FileReader.readAsArrayBuffer`, parse synchronously on main thread with progress callback | FIT files are typically < 10 MB; sync parsing is fast enough; avoids worker overhead | Web Worker (adds messaging complexity, justified only if files > 50 MB) |

## Migration Plan

### Cleanup after migration

A **"Purge legacy tables"** button on the Settings page calls `db.deleteLegacyTables()`, which drops old `activities` and `activityDetails` tables by clearing them and removing references. Button is disabled until migration is confirmed (user can verify data looks correct first). Shows confirmation dialog before executing. Frees browser storage once user is satisfied.

### Version 8 upgrade callback (runs automatically when user loads the app)

1. **Read all old `activities`** — iterate, generate UUID per row, insert into `allActivities` with `source = 'strava'` and `externalId = old numeric ID`
2. **Build ID map** — `Map<number, string>` (old numeric ID → new UUID)
3. **Read all old `activityDetails`** — iterate, look up UUID from ID map, insert into `allActivityDetails`
4. **Read old FK tables** — `segments`, `segmentEfforts`, `routeGroups`, `routeActivities`, `activitySegments`. For each row, replace `activityId` with the UUID from the ID map. Insert into the redefined v8 tables. For `segmentEfforts`, also build a segment ID map since `segments.id` gets re-generated by auto-increment.
5. **Remove legacy tables** (optional) — old `activities` and `activityDetails` tables remain (Dexie ignores unlisted tables).

### URL backward compatibility

- `ActivityDetail.tsx` checks if URL param `:id` is numeric: if so, treat as `externalId` lookup in `allActivities` and redirect to UUID-based URL
- After first redirect, all links use UUIDs

### Rollback

- Old `activities` and `activityDetails` tables are preserved in the v8 schema (data untouched). If rollback needed, a v9 could reverse the copy.

### Legacy table cleanup

- Not automatic — user must manually trigger cleanup via Settings button. Reduces risk: if migration has issues, old data is still available.
- Dexie cannot DROP tables at runtime; cleanup clears old tables and removes them from future version schemas.
- Storage freed: old `activities` + `activityDetails` data (typical user: 100–200 KB).

## Risks / Trade-offs

- **[Migration] Large data sets may cause slow first load** → Mitigation: migration runs in a single Dexie transaction; show a "migrating data…" splash if > 1000 activities detected
- **[Migration] FK tables have auto-increment IDs that shuffle** → Mitigation: segment/effort ID maps are built in the upgrade callback; all cross-references are remapped correctly
- **[Performance] Large FIT files (> 50 MB) block the UI during parsing** → Mitigation: Show progress spinner per file; future optimization can add Web Workers if needed
- **[Compatibility] Not all FIT file versions/features are covered by fit-file-parser** → Mitigation: Graceful error handling per file; invalid files show error status without crashing the batch
- **[PWA] Share target only works on Android Chrome / Chromium browsers** → Mitigation: Fallback to manual file upload on unsupported platforms; feature-detect before showing share instructions
- **[Storage] Dexie has no size limit but browser IndexedDB quotas vary** → Mitigation: Show storage usage estimate before import; warn if approaching quota (~ 50 MB free)
- **[Breaking] Existing bookmarks with numeric activity IDs break** → Mitigation: Numeric ID detection + redirect to UUID URL ensures old links still work
